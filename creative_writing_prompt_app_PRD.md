# Product Requirements Document — Morning Craft Prompt App

*Working title: TBD (placeholder: "Longhand")*

## 1. Vision

A quiet, daily writing tool that delivers one curated craft exercise each morning,
gives the writer a frictionless place to respond, and files the result into Google Docs
where it can later be organized by project or theme. Over time the writer accumulates a
corpus of emotionally rich, craft-deliberate fragments that can be folded into larger
work. The tool's job is to **start the morning and stay out of the way.**

## 2. Defining principle: minimal AI footprint

This is the constraint that governs every other decision. The writer is explicitly wary
of "AI fingerprints" on the work and of AI interfering with the creative process.
Therefore:

- The app **never generates, rewrites, critiques, or edits the writer's prose.**
- Prompts are served from a **fixed, human-authored library** — not generated by an LLM at runtime.
- Tagging is **derived from prompt metadata, not from reading the writer's text** (see §6),
  so no model ever needs to look at what the writer wrote.
- Net effect: the running application can function with **zero LLM calls.** AI (Claude Code)
  is used only to *build* the tool, never to operate inside the creative loop.

Any future AI-assisted feature (e.g., theme detection across entries) must be strictly
opt-in, off by default, and clearly quarantined from the writing surface.

## 3. Target user & context

- A single writer (the owner), new-to-intermediate in craft, working alone.
- Used first thing in the morning, primarily on **mobile and desktop web, weighted equally** —
  the app must be fully responsive and pleasant to write in on a phone.
- Sessions are usually **deep (30+ minutes)** on good days, but the tool must also rescue
  **hard mornings** where getting started is the whole battle (see §5.3).

## 4. Craft taxonomy

Every prompt belongs to exactly one craft area. The library covers all four:

1. **Character & interiority**
2. **Sensory detail & emotional description**
3. **Dialogue & voice**
4. **Theme & structure / arc**

These map to the methods in the recommended craft texts (Novakovich, LaPlante, Le Guin,
Burroway, and the broader MFA-workshop tradition). **Important:** library prompts must be
*original exercises written in the spirit of* these methods — not verbatim copies of any
book's exercises — to keep the library clean of copyrighted material.

## 5. Core features

### 5.1 Daily prompt
- On open, the app surfaces **one** prompt for the day, chosen by the rotation logic (§7).
- Prompt is shown with: the exercise text, its craft area, an estimated depth, and a one-line
  "what this trains" note.
- The writer can **swap** for a different prompt (same area or any area) if today's doesn't land.

### 5.2 Writing surface → Google Docs
- Per the writer's choice, **Google Docs is the system of record for prose.**
- On starting a session, the app creates (or appends to) a Google Doc pre-seeded with the
  date, the prompt, and the craft tag as a header, then opens it for writing.
  - **Default decision (flag for confirmation):** one Google Doc *per entry*, collected in a
    dedicated Drive folder, rather than one giant running doc. Easier to drag into project
    folders later. Easy to change.
- The writer composes directly in Google Docs (excellent mobile + desktop editor; nothing to rebuild).

### 5.3 Hard-morning mode ("Just start")
- A prominent low-friction option for days when starting is hard.
- Serves an extra-short, extra-gentle on-ramp (a 2–3 minute warm-up, e.g. a single sensory
  line or one concrete observation) with explicit permission to stop there.
- Goal: lower the activation energy to near zero; momentum is the only target.

### 5.4 Organize by project / theme
- Organization happens **in Google Docs / Drive** (the writer's chosen surface): they move or
  label docs into project or theme folders freely.
- The app's index (§6) records the craft tag and metadata so entries remain **searchable and
  filterable** from within the app, without duplicating Docs' organizing role.

## 6. Data model — the index (DB "just indexes")

The database is a lightweight **index only**. It never stores the prose (that lives in Google
Docs). One row per entry:

| field            | notes                                                        |
|------------------|--------------------------------------------------------------|
| `id`             | entry id                                                     |
| `date`           | session date                                                 |
| `prompt_id`      | FK into the prompt library                                   |
| `craft_area`     | derived from the prompt — **no text analysis required**      |
| `google_doc_id`  | the Drive doc holding the prose                              |
| `google_doc_url` | deep link for one-tap reopen                                 |
| `project_label`  | optional, writer-supplied                                    |
| `word_count`     | optional, fetched from Docs metadata                         |
| `completed`      | did they write, or just open                                 |

Tags = `craft_area` (automatic, from prompt metadata) + optional `project_label` (manual).
**No model ever reads the writer's words to produce a tag.**

A second table holds the **prompt library** itself:

| field          | notes                                              |
|----------------|----------------------------------------------------|
| `id`           | prompt id                                          |
| `craft_area`   | one of the four                                    |
| `text`         | the exercise                                       |
| `trains`       | one-line "what this builds"                        |
| `depth`        | quick / focused / deep                             |
| `order_hint`   | rough foundational→advanced progression within area|
| `lineage`      | which craft tradition it echoes (for the writer's interest) |

The library ships as an editable seed file (JSON) in the repo so it can be version-controlled
and expanded by hand over time.

## 7. Prompt rotation logic (deterministic, no AI)

- Maintain a pool of unused prompts per craft area.
- Each morning, default behavior: **rotate the craft area** (Character → Sensory → Dialogue →
  Theme → repeat) so all four get exercised over a cycle; pick a not-recently-used prompt from
  that area, gently respecting `order_hint` (foundational before advanced).
- Writer can override: pick a specific area, reshuffle, or "give me anything."
- When a pool empties, reset it. No adaptation, no scoring, no model in the loop.

## 8. Architecture & stack (recommended defaults — all changeable)

Optimized for equal mobile + desktop web use.

- **Frontend:** React + Vite, responsive layout (Tailwind). Must be genuinely writable on a phone.
- **Backend:** Node + Express (thin) — handles Google OAuth 2.0 and proxies the Google Docs +
  Drive API calls; serves prompts from the library.
- **Index DB:** SQLite (via Prisma or better-sqlite3) — trivial for a single-user personal tool.
  Swap to Postgres only if multi-device hosted sync is wanted later.
- **Auth / integration:** Google OAuth 2.0; Google Docs API (create/seed docs) + Drive API
  (folder placement, metadata). Requires the writer to register Google API credentials once.
- **AI at runtime:** none. (Claude Code is the build tool, not a dependency of the running app.)

## 9. Non-goals (explicitly out of scope)

- No AI critique, scoring, or feedback on the writer's prose.
- No AI co-writing, autocompletion, or "improve this" anywhere near the writing surface.
- No AI-generated prompts at runtime (library is human-authored).
- No social, sharing, gamification, or streak-pressure mechanics.
- No analysis of the writer's text to infer themes (deferred; opt-in only if ever built).

## 10. Open decisions (defaults chosen; confirm or override)

1. **One doc per entry vs. one running doc** — defaulted to per-entry in a Drive folder.
2. **Craft-area rotation vs. free choice as the default each morning** — defaulted to rotation,
   with override always available.
3. **Library size for v1** — suggest ~10–15 original prompts per area (40–60 total) to start.
4. **Hosting** — run locally for v1; decide on deployment later (Google OAuth redirect URIs
   will need updating when you do).
5. **Title / branding.**

## 11. Suggested build phases for Claude Code

- **Phase 1 (MVP):** React shell (responsive) + SQLite index + seed prompt library + daily
  rotation + "show today's prompt" + manual "I wrote in this doc, here's the link" capture.
  No Google API yet — prove the loop.
- **Phase 2:** Google OAuth + auto-create/seed the daily Google Doc + store doc id/url in index.
- **Phase 3:** Hard-morning mode, prompt swapping, project labels, in-app search/filter over the index.
- **Phase 4 (optional):** Drive folder auto-placement, word-count metadata, library editor UI.
